`example_index=0` and crashed the coordinator with an uncaught
`ValueError` after both workers completed process execution -- root

ever be constructed. Full failure analysis, the coordinator/verifier
repairs made before B2A-R2, and the pre-registered, deterministic,

`docs/B2A_R1_FAILURE_AND_B2A_R2_PROTOCOL_2026-07-22.md`.

## 5. Two verification-layer defects found and repaired (not scientific)

This was the first real execution ever to reach this deep into a
genuinely-eligible R-KV example in this harness (B2A-R1 crashed before
producing comparable evidence), and it surfaced two pre-existing,
previously-unexercised defects in the coordinator's verification code --
neither affects the scientific result above, and neither was weakened to
force a pass:

1. **`single_rtx3090_verified` false positive-negative.**
   `cli_device_preflight`'s `compute_capability` is the native `(8, 6)`
   tuple `StrictDeviceEvidence` produces in-process, while
   `fullkv.device_evidence`/`rkv.device_evidence` round-trip through each
   worker's JSON result file and come back as the list `[8, 6]` --
   `(8, 6) == [8, 6]` is `False` in Python even though every one of the
   three independent hardware observations (CLI, FullKV worker, R-KV
   worker) genuinely agreed on one RTX 3090. Fixed by normalizing
   list/tuple values before comparison in
   `verify_device_gate_from_raw_evidence`; a genuine mismatch (different
   capability, different length) still fails.
2. **`attempt_artifacts_verified` false negative.**
   `kvcot.discovery.orchestrator.run_example`'s `operation_runner` calls
   genuinely, durably emit `"pass1_plan_construction"` and
   `"minimized_target_evidence_construction"` progress-journal stages
   (confirmed directly against that module's source) that
   `RKV_KNOWN_PROGRESS_STAGES` had never been extended to recognize. Fixed
   by adding both as known R-KV stages.

Both fixes were verified by replaying them against this exact,
already-collected, unmodified attempt evidence -- no new GPU inference was
needed or performed to confirm them. Regression tests:
`tests/unit/discovery/test_strict_device.py`,
`tests/unit/discovery/test_attempt_verification.py`.

## 6. Verdict

**B2A-R2 FINAL VERDICT: FAIL -- B2B BLOCKED.**

A row qualified and the calibration ran to completion with every
scientific, parity, identity, event/pair-count, and memory condition
passing -- the sole failure is a resource-projection gate: the frozen
one-example row that was long enough to produce real R-KV eviction
(4822 FullKV tokens, 22 measured compaction events) projects to 5.01
GPU-hours for a full 12-example/144-real-pair pilot, 25% over the 4.00-hour
limit. This is an inherent tension in the qualification criteria (a row
needs enough length to trigger real events, but enough length drives up
the per-example and projected-pilot cost) -- not a software defect, not
fixed by retrying, and not something this document weakens or reruns
around.

## 7. Limitations

- No B2A-R2 scientific claim is made; the attempt failed its runtime gate.
- No B2B (12-example pilot) has run or is authorized by this result.
- No FaithKV method exists in this repository.
- This is the single authorized B2A-R2 attempt; inference began and it is
  now consumed. No further attempt is authorized by this document.
- The frozen-row scientific eligibility for a bounded-runtime pilot
  (a row long enough to trigger real events but short enough to project
  under 4.00 GPU-hours across 12 examples) remains an open question for
  any future, separately-authorized attempt.
