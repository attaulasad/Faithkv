
### H3 — CONFIRMED
> `compression/r1_kv.py` selects `budget - window_size` old slots per KV head and appends the most
> recent `window_size` slots — i.e. budget is the post-compaction cache length and includes the
> recent window.

`HuggingFace/rkv/compression/r1_kv.py:82` and `:168-179`:
```python
# shape: (bsz, num_kv_heads, budget - window_size)
indices = final_score.topk(self.budget - self.window_size, dim=-1).indices
...
indices = indices.unsqueeze(-1).expand(-1, -1, -1, head_dim)
k_past_compress = key_states[:, :, : -self.window_size, :].gather(dim=2, index=indices)
v_past_compress = value_states[:, :, : -self.window_size, :].gather(dim=2, index=indices)
k_cur = key_states[:, :, -self.window_size :, :]
v_cur = value_states[:, :, -self.window_size :, :]
key_states = torch.cat([k_past_compress, k_cur], dim=2)
value_states = torch.cat([v_past_compress, v_cur], dim=2)
return key_states, value_states
```
`(budget - window_size)` selected old slots + `window_size` most-recent slots = exactly `budget`
slots post-compaction. Confirmed exactly as stated. Additionally: `update_kv` is a **no-op**
(`r1_kv.py:46-47`, `if kv_cache_len < self.budget: return key_states, value_states`) whenever the
current cache length is still below budget — eviction only fires once the cache has actually
reached `budget` tokens.

### H4 — CONFIRMED, with an important refinement
> With `divide_method=step_length`, the patched CausalLM schedules compression periodically using
> `divide_length` (paper-style value 128), so instantaneous cache length can exceed `budget` by
> tokens accumulated since the last compaction.

`HuggingFace/rkv/modeling.py:555-563` and `:598-611`:
```python
if len(past_key_values) == 0:
    if self.config.compression_content == "think":
        self.after_think = False

if not hasattr(self, "length"):
    self.length = input_ids.shape[1]
else:
    self.length += input_ids.shape[1]
...
elif self.config.divide_method == "step_length":
    is_newline = self.length % self.config.divide_length == 0
...
if self.config.compression_content == "think" and self.after_think == True:
    is_newline = False

for layer in self.model.layers:
    layer.self_attn.config.compression = is_newline
```
`self.length` is a **cumulative absolute-token counter attached to the top-level `CausalLM`
object** (not the cache, not a layer), incremented once per top-level forward call by
`input_ids.shape[1]`. The compression flag `is_newline` becomes `True` exactly when `self.length`
is a multiple of `divide_length`, and this flag is set **after** computing the current forward's
logits — so it takes effect starting with the *next* top-level forward call, not the one that
crossed the multiple.

**Refinement beyond the hypothesis, confirmed from the same code plus
`results/validation-2026-07-02-a100/README.md` in the pinned tree**: because R1KV's own eviction is
itself a no-op below `budget` (H3), the *first* real eviction does not happen at `divide_length`
tokens — it happens at the first `divide_length`-multiple **at or after** the cache has reached
`budget` tokens. The validation README states this explicitly: "FullKV and R-KV generations are
token-identical until the first compression event (trigger ≈ budget + buffer ≈ 1152 tokens)" for
`budget=1024`, i.e. trigger ≈ budget + divide_length. Between compaction events, physical cache
length grows unboundedly (up to `budget + divide_length - 1` in the worst case) before the next
scheduled check fires and evicts back down to exactly `budget`. This is exactly the behavior §9's
`tokens_since_last_compaction` / `instantaneous_retention_ratio` fields are designed to capture.

## 3. Independent audit questions (not from H1–H8)

### 3.1 Exact eviction schedule and observation-window mechanism
Two independent window concepts share the same `window_size` value (frozen at 8 in this repo, per
§4), confirmed at `modeling.py:259-284` and `r1_kv.py:37-180`:
- **Query observation window**: the most recent `window_size` *query* vectors per layer, cached on
  `past_key_value.query_cache[layer_idx]`, used as the scoring queries against which historical
  keys are attention-scored (`r1_kv.py:49`, `compute_attention_scores(query_states, key_states)`
  where `query_states` here is the cached window, not the single current-step query).
- **Retained recent K/V window**: the most recent `window_size` key/value slots, always kept
  verbatim without competing in the topk selection (`r1_kv.py:176-177`).

The **schedule** (when compression is attempted) is decoupled from the **trigger** (whether it
actually evicts): schedule = `self.length % divide_length == 0` (`modeling.py:601`), trigger =
`kv_cache_len >= budget` inside `R1KV.update_kv` (`r1_kv.py:46`). See H4's refinement above for the
resulting sawtooth cache-length pattern.

### 3.2 Mutable state living outside the KV tensors
Enumerated by direct reference, for the reset function required in §3.1 of the brief:
- `self.length` — int, on the `CausalLM` model object (`modeling.py:560-563`). Cumulative absolute
  token count across the whole generation; must be deleted/reset before each independent run.
- `self.after_think` — bool, on the `CausalLM` model object, **only when
  `compression_content == "think"`** (`modeling.py:556-559, 593-596`). Not used in this repo since
  §4 freezes `compression_content=all`, but the attribute existing conditionally means our reset
  function must not assume its presence.
- `layer.self_attn.config.compression` — **tri-state** (`None` / `True` / `False`), one flag per
  attention layer, stored on that layer's `self_attn.config` (`modeling.py:230-235` sets initial
  value from `compression_config["compression"]`, which `run_math.py:230` sets to `None`;
  `modeling.py:609-610` overwrites it every top-level forward call). Each layer holds its own
  `Qwen2Config` instance (constructed per-layer by `Qwen2Attention_init`, not a single shared
  config object), so the flag must be reset on **every layer individually** — confirmed by the
  explicit `for layer in self.model.layers:` loop upstream itself uses to set it
  (`modeling.py:609-610`).
- `past_key_value.query_cache` — dict keyed by `layer_idx`, dynamically attached to the Cache
  object (`modeling.py:260-261`). A fresh Cache object naturally has none of this state, so
  "construct a fresh cache" (§3.1 of the brief) is sufficient for this one, provided a *new* Cache
  instance is actually constructed each run (not a cleared/reused one).
- Per-layer `kv_cluster` (`R1KV`) instance state, only when `record_kept_token_indices=True`:
  `evicted_token_num`, `kept_token_indices`, `kept_attention_scores`, `kept_similarity_scores`,
  `kept_final_scores` (`r1_kv.py:29-35`). One `R1KV` instance exists **per layer**
  (`modeling.py:231-234`, constructed inside `Qwen2Attention_init` which runs once per layer), so
  these lists are already layer-scoped, but they persist and grow across an entire generation and
  must be cleared before each independent run.
- `model.newline_token_ids`, `model.after_think_token_ids` — set once by `run_math.py:271-282`
  (not part of `rkv/`), static vocabulary-derived token-id lists. Not per-example state; do not
  need per-run reset, but this repository must set them itself since it does not call
  `run_math.py`.

Not upstream state, but a consequence worth flagging for our own runtime: `self.config.update(...)`
in `Qwen2Attention_init` (`modeling.py:230-231`) calls `PretrainedConfig.update()` which mutates
the config object **in place** — the `compression_config` dict values (`method_config`,
`compression`, `update_kv`) become permanent attributes on each layer's config object for the
lifetime of that layer object, not just for one call.

### 3.3 Can prompt tokens be evicted, or only generated tokens?
**Prompt tokens can be evicted**, including during the very first forward call (prefill), before
any token has been generated. `modeling.py:288-308` (the `if self.config.compression is None:`
branch, which is the state on the *first* top-level forward call since `compression_config`
initializes it to `None` and it is only overwritten after that call completes):
```python
if self.config.compression is None:
    key_states_compress, value_states_compress = self.kv_cluster.update_kv(
        key_states,       # <-- this forward's freshly-projected K, i.e. the WHOLE prompt on prefill
        cached_queries,    # last `window_size` queries of the prompt (set at modeling.py:266-271)
        value_states,
    )
    if self.config.update_kv is True:
        past_key_value.update(key_states_compress, value_states_compress, self.layer_idx, cache_kwargs)
```
On prefill, `key_states` has not yet been written to the cache — it is the raw projection of the
entire prompt for this layer. If the prompt is at least `budget` tokens long, `R1KV.update_kv` will
evict prompt tokens right here, and **only the already-compressed result is ever stored** in the
cache (the uncompressed prompt K/V is never cached when `update_kv=True`, which is this
repository's frozen setting). This directly confirms §3.4's requirement to track prompt-token vs.
reasoning-token retention separately, and confirms the requirement not to pin the prompt (doing so
would depart from this exact published code path).

### 3.4 Per-layer/per-KV-head kept-index information without altering scores/selections
Confirmed obtainable — see H5's second paragraph above. `record_kept_token_indices=True` only adds
extra tensor bookkeeping computed from the *same* `indices` used for the real eviction; it does not
change `topk` inputs, outputs, or the gather that actually mutates the cache. Our replay/provenance
layer (§6 of the brief) should still maintain its own independent absolute-position tracking rather
than relying solely on these lists, per H5 (events-only, not continuous).

### 3.5 True installable dependency range
Confirmed identical in both `HuggingFace/requirements.txt` and `HuggingFace/pyproject.toml`:
`transformers>=4.48.1,<4.56`. Both files carry the same explanatory comment: the monkeypatch is

### 3.6 Cache-internals coupling (volunteered finding, not asked for but load-bearing for §3.2/§6)
The `elif self.config.compression is True:` branch does **not** use the Cache object's public
`.update()` return value for the compressed write-back; it reaches into and overwrites the
`DynamicCache` implementation's private list attributes directly:
```python
past_key_value.key_cache[self.layer_idx] = key_states_compress
past_key_value.value_cache[self.layer_idx] = value_states_compress
```
(`modeling.py:323-325`, Qwen2 variant; identical pattern at lines 166-168 and 491-492 for
Llama/Qwen3). This is a second, independent source of the `<4.56` ceiling: `key_cache`/`value_cache`
as public-ish list attributes on `DynamicCache` are themselves an internal-implementation detail of
`transformers`'s Cache class that can be renamed or restructured across versions. Any GPU
validation test that inspects cache shapes/contents must go through the same attribute names this
patch uses, not assume a stable Cache API.

```python
    ).mean(dim=-2).to(query_states.dtype)
)                                                                                      # lines 51-59
attn_cache = F.max_pool1d(attn_weights_sum, kernel_size=self.kernel_size,
                           padding=self.kernel_size // 2, stride=1)                    # lines 62-67
similarity_cos = cal_similarity(
    key_states, retain_ratio=self.retain_ratio, retain_direction=self.retain_direction
)[:, :, : -self.window_size]                                                           # lines 69-73
final_score = attn_cache * self.mix_lambda - similarity_cos * (1 - self.mix_lambda)    # lines 75-77
indices = final_score.topk(self.budget - self.window_size, dim=-1).indices             # line 82
```

**`mix_lambda` is a single linear blend coefficient, not a gate or a two-stage filter.**
`final_score` is one combined tensor: the attention-importance term `attn_cache` weighted by
`mix_lambda`, minus the redundancy term `similarity_cos` weighted by `(1 - mix_lambda)`. `topk`
(line 82) runs **once**, directly on this already-blended `final_score` — there is no separate
"first filter by similarity, then rank by attention" pipeline, and no separate topk call for the
similarity component. Because `similarity_cos` is *subtracted*, a token that is highly redundant
with other kept tokens gets a *lower* `final_score` and is *less* likely to survive the top-k
selection at a fixed `mix_lambda`; `mix_lambda=0.1` (frozen) means the redundancy term is weighted
9× more heavily than the raw attention term in this blend (`0.9` vs `0.1`).

### 7.2 Where `retain_ratio` and `retain_direction` are consumed

Entirely inside `cal_similarity`, called from r1_kv.py:69-73 (and again at r1_kv.py:91-95 for the
analysis/recording path only — same computation, not a second live decision).
`HuggingFace/rkv/utils.py:42-86`:
```python
def cal_similarity(key_states, threshold=0.5, retain_ratio=0.2, retain_direction="last"):
    ...
    similarity_mask = similarity_cos > threshold                                       # line 56
    k = max(1, int(seq_len * retain_ratio))                                             # line 58
    indices = torch.where(similarity_mask, torch.arange(seq_len, ...), torch.zeros_like(...))
    if retain_direction == "last":
        similarity_retain = torch.max(indices, dim=-1)[0]                               # line 68
    elif retain_direction == "first":
        similarity_retain = torch.min(indices, dim=-1)[0]                               # line 72
    elif retain_direction == "last_percent":
        similarity_retain = torch.topk(indices, k=k, dim=-1)[0][:, :, 0]                # line 76
    elif retain_direction == "first_percent":
        similarity_retain = torch.topk(indices, k=k, dim=-1, largest=False)[0][:, :, -1] # line 80
    similarity_cos.scatter_(-1, similarity_retain.unsqueeze(-1), 0)                      # line 84
    return similarity_cos.mean(dim=-2).softmax(dim=-1)
```
`retain_ratio`/`retain_direction` are fully resolved **inside** `cal_similarity`, i.e. **before**
`mix_lambda` ever combines anything — they shape the `similarity_cos` component that `final_score`
later blends with `attn_cache`. There is no separate post-topk redundancy filter and no
pre-topk hard cutoff by ratio; `cal_similarity`'s output is a dense per-token redundancy score
(softmax-normalized), not a binary keep/drop mask, and it enters `final_score` as one continuous
additive term, not a filter stage.

Mechanically: for each pair of tokens whose cosine similarity exceeds `threshold` (default `0.5`,
**not** exposed as a frozen or CLI parameter — `r1_kv.py` never passes `threshold` to
`cal_similarity`, so it is fixed at the hardcoded default regardless of any setting in this
repository's configs), `retain_direction` picks **one** "protected" partner index per row
(the most-recent similar token for `"last"`, the earliest for `"first"`, or a `retain_ratio`-sized
percentile position for `"last_percent"`/`"first_percent"`). That protected pair's similarity entry
is zeroed (`scatter_`, line 84) before the row-wise mean, which slightly *reduces* the redundancy
penalty attributed to whichever token was chosen as the "keeper" of a near-duplicate cluster.

### 7.3 Load-bearing finding: `retain_ratio` is inert under the frozen `retain_direction=last`

`k = max(1, int(seq_len * retain_ratio))` (utils.py:58) is computed unconditionally, but **only
read by the `"last_percent"` (line 76) and `"first_percent"` (line 80) branches**. The `"last"`
branch (utils.py:67-68) — this repository's frozen `retain_direction` value — calls
`torch.max(indices, dim=-1)[0]` with no reference to `k` anywhere in that branch or afterward.

**Consequence for this repository: under the frozen configuration (`retain_direction=last`,
`retain_ratio=0.2`), the value of `retain_ratio` has zero effect on R-KV's eviction behavior.** It
is computed and then never used. This is not a bug in this repository — it is upstream's actual
behavior at the pinned commit, confirmed by reading the branch dispatch directly — but it means:

- Do not describe `retain_ratio=0.2` in any results/docs as a tunable that shaped the reported
  operating point; under `retain_direction=last` it is inert.
- If a future `CHANGELOG.md`-gated change ever switches `retain_direction` to `"last_percent"` or
  `"first_percent"`, `retain_ratio` becomes live and this note must be revisited.
- The frozen-settings table in `configs/lock.yaml` and `docs/EXPERIMENT.md` should carry a comment
  citing `utils.py:58-80` so a future reader doesn't waste time tuning a dead parameter.
